## Important Working Principles Going Forward

This section is intentionally explicit because the repository repeatedly ran into the same failure mode.

### A. Do not solve regressions by whack-a-mole blocking

Avoid:

- one-off phrase blockers
- “special handling” layered on top of unclear architecture
- defensive fences around unknown routes

Prefer:

- removing the route entirely
- reducing write authority to fewer places
- documenting the intended authority boundary clearly

### B. Do not declare a route removed by intuition

Before saying “this old path is gone,” verify it.

That means:

1. identify all files that can mutate or reshape the target state
2. search the repo
3. distinguish:
   - active runtime path
   - backward-compatibility-only path
   - dead/unreachable residue
4. only then conclude removal

### C. File splitting is not enough

We learned the hard way that:

- splitting files improves visibility
- but does **not** automatically improve authority boundaries

When refactoring, always ask:

- is this file the right owner of this behavior?
- is the import/export surface necessary and sufficient?
- does this wiring create a future hidden mutation path?

If the answer is “probably not,” clean it immediately rather than leaving residue for later.
