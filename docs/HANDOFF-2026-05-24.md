## Completed In This Session

### Multi-Recipient Kin Send

The Kin list now has two separate controls:

- `送信対象`: one or more Kin profiles that receive normal user messages and
  ordinary `SYS_INFO` shares
- `タスク実行`: a single Kin profile used for task execution and task protocol
  replies

`ALL` in the Kin list is now a toggle. It selects all Kin as `送信対象`, and
when all Kin are already selected it clears the selection.

When multiple Kin are selected for sending, the user message is displayed once
in chat instead of once per recipient. Kin replies still display with their
speaker labels.

### Protocol Routing Rules

Normal messages and `SYS_INFO` are multi-recipient sends.

Task-start and task-response protocol blocks are routed only to the
`タスク実行` Kin:

- `SYS_TASK`
- `SYS_TASK_CONFIRM`
- `SYS_USER_RESPONSE`
- `SYS_GPT_RESPONSE`
- `SYS_SEARCH_RESPONSE`
- `SYS_YOUTUBE_TRANSCRIPT_RESPONSE`
- `SYS_LIBRARY_DATA_RESPONSE`
- `SYS_LIBRARY_IMAGE_DATA_RESPONSE`
- `SYS_DRAFT_PREPARATION_RESPONSE`
- `SYS_DRAFT_MODIFICATION_RESPONSE`
- `SYS_PPT_DESIGN_RESPONSE`
- `SYS_FILE_SAVING_RESPONSE`

Task-context multipart delivery still routes only to the task-execution Kin.

### Multipart `SYS_INFO`

Multipart `SYS_INFO` can now be sent to multiple selected Kin.

Important behavior:

- each PART is sent to all selected Kin
- the next PART is set only after every selected Kin returns a receipt
- `Received.` and `Received. Send the next.` are both valid receipts
- multipart PART sends are sequenced across targets instead of fully parallel,
  avoiding dropped sends from external API concurrency pressure
- if a target does not return a usable response, the chat shows a failure line
  with that Kin's speaker label and the next PART is not advanced

### Kin-to-Kin Chat Isolation

Kin-to-Kin chat remains isolated from the normal multi-recipient send selection.
It continues to call `sendKinToKinMessage(kinId, text, speakerLabel)` with the
drawer-selected Kin id and does not read `selectedKinIds`.
