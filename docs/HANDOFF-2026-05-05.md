# Handoff 2026-05-05

## Active Entry Point

Start the next session by reviewing the next PPT generation result from the
image-library adaptive-layout work. The expected first user action is to run a
new `/ppt` design and PPTX output, then share the design text and PPTX for
review.

Review the result in this order:

1. PPT design document correctness before renderer behavior.
2. Image-library ID selection and per-image labels.
3. Slide-role decisions: `textMain` vs `visualMain`.
4. Adaptive text-main layout: text box size, image count, and non-overlap.
5. Bookends: cover containment/transparency and closing-slide duplication.
6. PPTX XML/media only after the design document route has been checked.

Do not start by patching renderer symptoms. The root-cause priority is upstream:
the PPT design document must not invent image IDs, mismatch image labels, or
create duplicate summary structure.

## Completed In This Slice

### Image-Library Planning

- `buildPresentationImageLibraryContext` now exposes a `Caption seed` for each
  candidate image.
- The prompt now states that `visualRequest.labels` must match
  `candidateImageIds` exactly:
  - same count
  - same order
  - one label per selected image ID
- Labels must be derived from that exact candidate's caption seed, title,
  visible subjects, or embedded text items.
- Planned process names must not be assigned to a photo unless that exact photo
  visibly contains the process.
- If the label is uncertain, the model should omit labels rather than guess.

Primary files:

- `lib/app/presentation/presentationImageLibrary.ts`
- `lib/app/presentation/presentationImageLibrary.test.ts`
- `lib/task/taskProtocol.ts`
- `lib/task/taskProtocol.test.ts`

### PPT Design Normalization

- `visualRequest.labels` is now accepted only when it can be aligned to image
  IDs. If `candidateImageIds` has three IDs and `labels` has four values, the
  labels are dropped instead of being indexed incorrectly.
- Multi-image expansion uses labels only when the count matches selected image
  IDs. Otherwise, image captions are hidden instead of reusing a generic or
  misaligned label.
- The PPT task prompt now tells the LLM not to use `summaryClosing` when the
  last body slide is already a summary, recap, conclusion, or future-outlook
  slide.
- Bookend normalization also enforces that rule by converting the closing slide
  to `endSlide` when the final body slide is already summary-like.

Primary files:

- `lib/app/presentation/presentationSlideFrames.ts`
- `lib/app/presentation/presentationTaskPlanning.ts`
- `lib/app/presentation/presentationTaskPlanning.test.ts`
- `lib/app/presentation/presentationPlanValidation.ts`
- `lib/app/presentation/presentationPlanValidation.test.ts`

### Text Encoding Cleanup

- PPT design formatting in `presentationTaskPlanning.ts` no longer contains
  mojibake string literals.
- Japanese display strings are either normal UTF-8 test expectations or escaped
  literals in source where that is safer.
- `check:utf8` passed after the cleanup.

### Renderer Safety

- Renderer V2 remains a verifier/safety layer, not the primary fix.
- Japanese text-height estimation for adaptive text-main layouts is now more
  conservative so image boxes are placed below or beside actual text capacity
  with more margin.
- Existing cover handling keeps `visualTitleCover` images contained and applies
  an 80% visual treatment so title text can coexist with the photo.

Primary files:

- `kin-presentation-renderer/src/rendererV2.ts`
- `kin-presentation-renderer/src/rendererV2.test.ts`

   PPTX, not only in XML geometry?
7. Are repeated generic image captions gone?
8. Are no mojibake strings present in the PPT design text?

image library candidate context -> LLM slideFrame JSON -> normalization ->
frameSpec -> renderer layout -> PPTX XML/media.

## Policy Note

This slice reinforced the engineering policy in
`docs/engineering-judgment-policy.md`:

- do not use brittle semantic heuristics for judgment-heavy behavior
- use LLMs with explicit staged logic
- keep fields necessary and sufficient
- do not use the user as first-line tester
- verify locally as far as practical before asking for another live generation
