## Session Summary

This session implemented the task-registration workspace and related Kin
protocol changes.

Implemented:

- Kin tab actions:
  - `レス共有` and `タスク共有` are now `SYS_INFO` sharing tools
  - `タスク共有` sends the full current task-formation content, not only an
    overview/key-point summary
  - `タスク登録` creates a task-registration draft instead of writing to Kin input
    or contaminating task formation
  - the unused response-receive path was removed from the UI/action wiring
- Task registration subtab:
  - registered task list
  - draft display
  - library reference settings
  - recurrence settings
  - multiple registered tasks with start/edit/delete
  - new registration / overwrite save / cancel
  - settings panels are hidden when no draft is active
- Registered task runtime behavior:
  - pressing start only sets the registered `SYS_TASK` into Kin input
  - task progress runtime starts only after that `SYS_TASK` is actually sent to
    Kin
  - matching is by `TASK_ID`
- Task-time library settings:
  - applied only after actual `SYS_TASK` send
  - previous library settings are restored when the active task reaches
    completed state
- Recurrence:
  - repeat tasks are checked every 30 seconds while the app is open
  - due tasks are placed into Kin input
  - existing auto-send settings can send the populated SYS input onward
  - mobile background execution is not part of this implementation
- `SYS_TASK_PROPOSAL`:
  - parsed from Kin replies
  - converted into the same task-registration draft path as user task
    registration
  - excluded from progress runtime ingestion
  - rulebook says Kin must wait for user approval followed by `SYS_TASK`

## Maintenance Recommendations

- Keep task registration draft state separate from task formation and active
  task runtime.
- Keep `レス共有` / `タスク共有` as `SYS_INFO`-only tools. User input should be a
  sharing directive, not a mode prefix.
- Keep `SYS_TASK_PROPOSAL` proposal-only. Never auto-start from proposal receipt.
- Treat recurrence as app-open scheduling until a deliberate background
  mechanism is designed.
- When touching registered-task start, verify both boundaries:
  - Kin input population
  - actual Kin send and `TASK_ID` matched runtime start
- Keep temporary task-time library overrides scoped to actual active task runs;
  do not persist draft-time settings into global library settings.
