### Trap To Avoid: `titleLineFooter` Duplication

The `titleLineFooter` duplication seen at closeout is not just a prompt wording
problem. The likely residue path was:

- `deckFrame.masterFrameId` correctly stores the common master.
- Some prompt examples still repeated `masterFrameId: "titleLineFooter"` inside
  each `slideFrame`.
- `formatPresentationSlideFramePlanLines` displayed both the common deck master
  and each slide master, so common values appeared duplicated in the visible PPT
  design text.
- Renderer schema defaults still use `titleLineFooter` when a master is omitted,
  which is correct for validation but can hide whether a slide intended an
  override.

Closeout cleanup performed:

- `lib/task/taskProtocol.ts` no longer shows `masterFrameId` inside the example
  slide frame.
- `lib/app/presentation/presentationTaskPlanning.ts` tells PPT design creation
  to omit slide-level `masterFrameId` unless it intentionally overrides the deck
  master.
- `lib/app/presentation/presentationSlideFrames.ts` no longer repeats the
  common master in visible slide lines. If the slide master equals the deck
  master, it shows `Frame: <layoutFrameId>`; if it differs, it shows
  `Frame: <masterFrameId> / <layoutFrameId>`.

Still open:

- `PresentationTaskSlideFrame.masterFrameId` is still required in the normalized
  in-memory type. That is acceptable as a parsed/normalized representation, but
  the authored JSON should treat slide-level master as an override.
- The future frame registry should make the inheritance rule explicit instead
  of relying on examples and display formatting.

### Implemented Slice: Google Drive Image Import

Google Drive image import is now connected to the shared image import flow used
by device import.

Current behavior:

- Image-only import creates an ordinary text library card and, when the image
  library save toggle is on, an image-library card.
- Image + same-name text import uses the text file as the image metadata source
  and saves only to the image library.
- When image-library save is off, image files do not create image-library cards.

The implementation remains one shared circuit for Drive and device imports:

1. Pick an image file from Google Drive.
2. Download it as a Blob/File.
3. Run the ordinary text-ingest path so the normal library card is created.
4. If `画像取込` is on, also save the same file into the image library.
5. If `画像+描写テキスト` is selected, reuse the ordinary text-ingest output as
   the image card Description instead of making a second OpenAI call.
6. Show the same completion/error message style as device image import.

This is not a separate feature path. Do not create a second Drive-only image
import implementation; route it through the shared image import flow.

## Implemented Slice: Image Library To PPTX

PPTX rendering now consumes images already saved in the image library. Keep this
section as the design contract for the implemented path.

### Command policy

Extend `/ppt` image metadata as follows:

```text
/ppt
Document ID: ppt_xxx
Images: off
Create PPT
```

No images. Existing assets and API generation are disabled.

```text
/ppt
Document ID: ppt_xxx
Images: on, library
Create PPT
```

Use only image-library assets. If no good match exists for a visual block, leave
that block without an image or use a non-image fallback.

```text
/ppt
Document ID: ppt_xxx
Images: on, API
Create PPT
```

Use only new API-generated images. Generated images must still appear in chat
with `Image ID`, and remain revisable/savable by image commands.

```text
/ppt
Document ID: ppt_xxx
Images: on
Create PPT
```

Default hybrid mode. Use matching image-library assets first, then generate
missing assets through the API.

Also accept:

```text
Images: on, library, API
```

as an explicit hybrid equivalent.

### Required metadata

Before image-library assets can be reliably placed in PPTX, image records should
store:

- `widthPx`
- `heightPx`
- `aspectRatio`
- `orientation`: `landscape`, `portrait`, or `square`
- file size
- MIME type
- Description

Description answers "what is this image?" Width, height, and orientation answer
"where can this image fit without awkward cropping?"

### Placement policy

Do not special-case only "landscape" because it was the first example. Treat
orientation as a general layout signal:

- landscape: good for top/bottom layouts, full-width visual bands, or wide
  visual blocks
- portrait: good for left/right layouts with a vertical visual column
- square: good for centered image cards, balanced two-column blocks, icons, or
  compact visual panels
- extreme aspect ratios: prefer `contain` over crop unless the slide frame
  explicitly allows cropping

The renderer should use semantic match first and geometric fit second:

1. compare slide/block `visualRequest` brief/prompt/type with image Description,
   file name, and prompt
2. reject obviously unrelated images
3. prefer compatible orientation/aspect ratio for the target block
4. choose a layout variant that respects the chosen image geometry

### Client/server boundary

Image library assets live in browser-side IndexedDB. The server render route
cannot fetch those assets by `Image ID` by itself.

Therefore the client-side `/ppt` render flow must:

1. identify candidate image-library cards from `referenceLibraryItems`
2. hydrate the needed image bytes from IndexedDB with `loadGeneratedImageAsset`
3. pass selected/candidate image assets to `/api/presentation-render` as base64
   plus metadata
4. let the server/renderer resolve assets according to the requested image mode

Avoid a server-only lookup design unless image storage is moved to a backend.

### Implemented components

- `parsePptCommand` resolves image mode as `off`, `library`, `api`, or
  `hybrid`.
- `presentationGptFlow.ts` hydrates browser-side image-library candidates before
  calling `/api/presentation-render`.
- `/api/presentation-render` accepts library image assets with base64 plus
  metadata.
- `lib/server/presentation/imageGeneration.ts` resolves visuals by mode:
  - off: strip/no assets
  - library: match from provided assets only
  - api: generate assets
  - hybrid: library first, API fallback
- Used/generated image assets are returned in the render response and selected
  image IDs are retained in the saved presentation plan.
- Renderer image placement now uses aspect-ratio-preserving `contain`.
