## Current Implemented State

### PPT design and rendering

- `/ppt` can create, revise, preview, and render presentation design plans.
- `Document ID` is visible and is the routing key for revisions and PPTX output.
- The canonical slide authoring direction is frame JSON. Avoid returning to the
  older "make natural language first, parse it into JSON later" loop.
- PPTX rendering can generate visual assets through the OpenAI image API when
  image generation is enabled.
- API-generated presentation images are shown in chat with `Image ID`, can be
  revised by `Image ID: ...`, and can be saved to the image library.

### Image generation and image library

- `Create Image:` creates an image independently of PPT flow.
- Generated images appear in chat first and are saved to the image library only
  when the user uses `Image ID: ...` plus `Save`.
- Image revisions create a new `Image ID`.
- Image library cards are separated from the ordinary reference library view.
- Image cards are excluded from ordinary library reference behavior.
- Image card text includes:
  - Image ID
  - file name
  - file size in KB with grouped digits, e.g. `1 788.6 KB`
  - original MIME
  - prompt / original prompt
  - usage for generated images
  - Description when available

### Image import

- Existing file/image/PDF text ingestion remains active for ordinary library
  cards.
- Library settings now have:
  - `テキスト取込方針`
  - unified ingest detail and character limit
  - an independent `画像取込` panel
- `画像取込` has:
  - image-library save on/off
  - `画像のみ`
  - `画像+描写テキスト`
- When image-library save is off, image files are imported only as ordinary
  text library cards.
- When image-library save is on, image files are also saved to the image
  library.
- When `画像+描写テキスト` is selected, Description reuses the ordinary library
  text ingestion result. It does not call OpenAI a second time.
- `/api/ingest` sends image uploads to OpenAI Responses as `input_image`, while
  PDFs and non-image files remain `input_file`.

## Next Feature: Image Library To PPTX

The next implementation should make PPTX rendering consume images already saved
in the image library.

### Command policy

Extend `/ppt` image metadata as follows:

```text
/ppt
Document ID: ppt_xxx
Images: off
Create PPT
```

No images. Existing assets and API generation are disabled.

```text
/ppt
Document ID: ppt_xxx
Images: on, library
Create PPT
```

Use only image-library assets. If no good match exists for a visual block, leave
that block without an image or use a non-image fallback.

```text
/ppt
Document ID: ppt_xxx
Images: on, API
Create PPT
```

Use only new API-generated images. Generated images must still appear in chat
with `Image ID`, and remain revisable/savable by image commands.

```text
/ppt
Document ID: ppt_xxx
Images: on
Create PPT
```

Default hybrid mode. Use matching image-library assets first, then generate
missing assets through the API.

Also accept:

```text
Images: on, library, API
```

as an explicit hybrid equivalent.

### Required metadata

Before image-library assets can be reliably placed in PPTX, image records should
store:

- `widthPx`
- `heightPx`
- `aspectRatio`
- `orientation`: `landscape`, `portrait`, or `square`
- file size
- MIME type
- Description

Description answers "what is this image?" Width, height, and orientation answer
"where can this image fit without awkward cropping?"

### Placement policy

Do not special-case only "landscape" because it was the first example. Treat
orientation as a general layout signal:

- landscape: good for top/bottom layouts, full-width visual bands, or wide
  visual blocks
- portrait: good for left/right layouts with a vertical visual column
- square: good for centered image cards, balanced two-column blocks, icons, or
  compact visual panels
- extreme aspect ratios: prefer `contain` over crop unless the slide frame
  explicitly allows cropping

The renderer should use semantic match first and geometric fit second:

1. compare slide/block `visualRequest` brief/prompt/type with image Description,
   file name, and prompt
2. reject obviously unrelated images
3. prefer compatible orientation/aspect ratio for the target block
4. choose a layout variant that respects the chosen image geometry

### Client/server boundary

Image library assets live in browser-side IndexedDB. The server render route
cannot fetch those assets by `Image ID` by itself.

Therefore the client-side `/ppt` render flow must:

1. identify candidate image-library cards from `referenceLibraryItems`
2. hydrate the needed image bytes from IndexedDB with `loadGeneratedImageAsset`
3. pass selected/candidate image assets to `/api/presentation-render` as base64
   plus metadata
4. let the server/renderer resolve assets according to the requested image mode

Avoid a server-only lookup design unless image storage is moved to a backend.

### Suggested implementation order

1. Add image dimensions to imported/generated image payloads and image card text.
2. Extend `parsePptCommand` from boolean `generateImages` to an image mode:
   `off`, `library`, `api`, `hybrid`.
3. Add tests for `Images: off`, `Images: on`, `Images: on, library`,
   `Images: on, API`, and `Images: on, library, API`.
4. In `presentationGptFlow.ts`, hydrate image-library candidates before calling
   `/api/presentation-render`.
5. Extend the presentation render request type to include library image assets.
6. In `lib/server/presentation/imageGeneration.ts`, replace the current
   generate-only hydration with a resolver:
   - off: strip/no assets
   - library: match from provided assets only
   - api: generate all needed assets
   - hybrid: library first, API fallback
7. Return all actually used/generated images in the render response so chat can
   show generated images and the stored presentation plan can retain selected
   image IDs.
8. Strengthen renderer image placement to use `contain` and aspect-ratio-aware
   frame variants.
