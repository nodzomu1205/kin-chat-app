### Non-Negotiable Debugging Rule

Do not present a plausible cause as the confirmed cause.

In this PPT renderer thread, a bad pattern appeared: a hypothesis such as
"the referenced image may have fallen outside the image-library reference
count" was treated too confidently before the actual evidence was checked.
The later screenshot showed a different, more concrete issue: the visible
library card had both an internal image ID and a title/alias that looked like
an image ID, and the PPT design had picked the alias. Future sessions must not
repeat the first pattern.

Required discipline for PPT design / renderer bugs:

1. Separate **observed facts**, **hypotheses**, and **confirmed causes** in the
   response.
2. Before claiming root cause, trace the value end to end:
   - visible library card metadata
   - PPT design `preferredImageId`
   - hydrated `libraryImageAssets`
   - resolver match result: exact ID / title alias / filename alias /
     semantic / generated / missing
   - final frameSpec passed to renderer
   - PPTX XML/media output
3. Do not hide unresolved routing bugs behind fallback behavior. Fallbacks are
   acceptable only after the primary route is made observable and deterministic.
4. If evidence is partial, say "partial verification" and name what is still
   unknown.
5. Add tests that reproduce the actual failure shape, not only tests for a
   convenient nearby failure.

## Implemented Slice: Image Library To PPTX

PPTX rendering now consumes images already saved in the image library. Keep this
section as the design contract for the implemented path.

### Command policy

Extend `/ppt` image metadata as follows:

```text
/ppt
Document ID: ppt_xxx
Images: off
Create PPT
```

No images. Existing assets and API generation are disabled.

```text
/ppt
Document ID: ppt_xxx
Images: on, library
Create PPT
```

Use only image-library assets. If no good match exists for a visual block, leave
that block without an image or use a non-image fallback.

```text
/ppt
Document ID: ppt_xxx
Images: on, API
Create PPT
```

Use only new API-generated images. Generated images must still appear in chat
with `Image ID`, and remain revisable/savable by image commands.

```text
/ppt
Document ID: ppt_xxx
Images: on
Create PPT
```

Default hybrid mode. Use matching image-library assets first, then generate
missing assets through the API.

Also accept:

```text
Images: on, library, API
```

as an explicit hybrid equivalent.

### Required metadata

Before image-library assets can be reliably placed in PPTX, image records should
store:

- `widthPx`
- `heightPx`
- `aspectRatio`
- `orientation`: `landscape`, `portrait`, or `square`
- file size
- MIME type
- Description

Description answers "what is this image?" Width, height, and orientation answer
"where can this image fit without awkward cropping?"

### Placement policy

Do not special-case only "landscape" because it was the first example. Treat
orientation as a general layout signal:

- landscape: good for top/bottom layouts, full-width visual bands, or wide
  visual blocks
- portrait: good for left/right layouts with a vertical visual column
- square: good for centered image cards, balanced two-column blocks, icons, or
  compact visual panels
- extreme aspect ratios: prefer `contain` over crop unless the slide frame
  explicitly allows cropping

The renderer should use semantic match first and geometric fit second:

1. compare slide/block `visualRequest` brief/prompt/type with image Description,
   file name, and prompt
2. reject obviously unrelated images
3. prefer compatible orientation/aspect ratio for the target block
4. choose a layout variant that respects the chosen image geometry

### Client/server boundary

Image library assets live in browser-side IndexedDB. The server render route
cannot fetch those assets by `Image ID` by itself.

Therefore the client-side `/ppt` render flow must:

1. identify candidate image-library cards from `referenceLibraryItems`
2. hydrate the needed image bytes from IndexedDB with `loadGeneratedImageAsset`
3. pass selected/candidate image assets to `/api/presentation-render` as base64
   plus metadata
4. let the server/renderer resolve assets according to the requested image mode

Avoid a server-only lookup design unless image storage is moved to a backend.

### Implemented components

- `parsePptCommand` resolves image mode as `off`, `library`, `api`, or
  `hybrid`.
- `presentationGptFlow.ts` hydrates browser-side image-library candidates before
  calling `/api/presentation-render`.
- `/api/presentation-render` accepts library image assets with base64 plus
  metadata.
- `lib/server/presentation/imageGeneration.ts` resolves visuals by mode:
  - off: strip/no assets
  - library: match from provided assets only
  - api: generate assets
  - hybrid: library first, API fallback
- Used/generated image assets are returned in the render response and selected
  image IDs are retained in the saved presentation plan.
- Renderer image placement now uses aspect-ratio-preserving `contain`.
- Frame-native PPTX output now routes through `kin-presentation-renderer/src/rendererV2.ts`.
  The V2 path applies the `titleLineFooter` master to visual-only slides,
  renders unresolved visual blocks as labeled placeholders, top-aligns text
  blocks, and expands `singleCenter` image placement to use the available frame.
- Explicit image-library IDs are strict: when `preferredImageId` is present and
  missing from the provided library assets, the resolver no longer substitutes a
  semantically similar but different image.
- Render input now sanitizes incomplete saved `visualRequest.asset` objects
  before renderer schema parsing. If a saved frame has `asset.imageId` but no
  `asset.base64`, the route removes that incomplete asset and restores the ID as
  `preferredImageId` so library hydration can resolve it deterministically.
- `normalizePresentationVisualMainPolicy` now collapses consecutive
  `singleCenter` visual-only slides that point at the same image ID, so a dense
  image is not repeated as two identical visual slides.
- After slide-frame normalization, `deckFrame.slideCount` is synchronized to
  the actual frame count. Do not paper over mismatched totals in the footer;
  fix the normalized deck model first.
- Text formatting is moving to block-style data, not renderer hard-coding.
  `renderStyle.textStyle` now carries heading/body/item font sizes, paragraph
  gaps, bullet indentation, and line spacing. Renderer V2 reads these style
  values; it should not become the place where one-off visual preferences are
  permanently baked in.
- The current default text block styles use `headingGapLines`, `bodyGapLines`,
  and `itemGapLines` of `2` for the main text/list styles. Keep these as
  block-style data so future decks can choose different body treatments without
  changing renderer code.
- `normalizePresentationVisualMainPolicy` now also performs final frame
  normalization even when no duplicate primary visual exists:
  - long slash-separated titles are compacted to the one-line title budget
  - dense `heroTopDetailsBottom` detail slides are converted to `leftRight50`
    when the bottom detail row would force excessive text shrinking
  - slide numbers are renumbered after these structural changes
  This is design normalization, not a renderer fallback. If a future result is
  wrong, trace the design frame data first before adding fallback behavior.
- Deck bookends are now modeled at `deckFrame.openingSlide` and
  `deckFrame.closingSlide`, not inside body `slideFrames`. Supported bookend
  frame IDs are `titleCover`, `visualTitleCover`, `endSlide`, and
  `summaryClosing`. Renderer V2 inserts enabled bookends before/after body
  slides and keeps `pageNumber.scope: "bodyOnly"` unnumbered on those bookends.
  Use `scope: "allSlides"` only when the user explicitly wants cover/end slides
  included in numbering.
- Bookend selection is normalized from body slide evidence:
  - if early body slides contain a representative visual, `titleCover` upgrades
    to `visualTitleCover` and carries a copied `openingSlide.visualRequest`
  - if the final body slide reads like a summary/next-priorities slide,
    `endSlide` upgrades to `summaryClosing` and uses the final list as concise
    closing bullets
  - explicit disabled bookends are respected
  - bookend visual assets are resolved/stripped through the same image-library
    resolver path as body slide visuals
