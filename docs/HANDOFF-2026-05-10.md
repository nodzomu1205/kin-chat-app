## Key Lessons

### 1. Never Treat Visible Text As The Whole Truth

Saved PPT designs have multiple projections:

- library card text
- `structuredPayload` presentation-plan JSON
- chat `[show]` / library show display text
- frameSpec used by PPTX rendering
- hydrated render payload with image bytes

A fix is incomplete if only one projection changes. The final edit-sync bug was
exactly this: JSON/PPTX was fixed first, but library show display still showed
a deleted visual label until `presentationSlideFrameFormatting.ts` learned
about `hiddenLabelSlotIds`.

### 2. Identify The Owning Boundary Before Patching

For PPT bugs, first classify the owner:

- generation prompt / schema
- plan validation/completion
- visual-slot resolution
- library edit text parser
- JSON merge
- chat display formatter
- frameSpec projection
- renderer

Do not patch a renderer symptom if the plan JSON is already wrong. Do not add
prompt pressure if a deterministic invariant can be validated in code. Do not
write a fallback if the primary route has not been traced.

### 3. Preserve User-Selected Visual IDs

When editing visual text, preserve:

- `preferredImageId`
- `candidateImageIds`
- selected `selectionMatches[].imageId`

Only rebase label/need text onto the existing selected match unless the user
explicitly clears or changes the visual selection.

### 4. Label Deletion Is Not The Same As Slot Deletion

A user may delete an in-image display label because it visually obstructs the
selected image. That must not remove:

- the visual slot
- the selected image ID
- the prompt/need that explains why the image belongs there

Current representation: keep the slot and selected image, and store the hidden
label intent in `visualRequest.renderStyle.hiddenLabelSlotIds`.

### 5. Refactor Only With A Recoverable Slice

This session repeatedly confirmed the user's warning from earlier failures:
pulling code apart without reference checks can collapse accumulated behavior.

Safe slice pattern used here:

1. state the touch boundary
2. run `rg` for references
3. move one pure helper family
4. keep compatibility or update only direct importers
5. run focused tests
6. run build
7. commit before the next slice

When `slidePartsParser.ts` initially looked risky because of string-literal
encoding, the work paused, checked source with `node`, and then proceeded only
with the safe display-helper extraction.
