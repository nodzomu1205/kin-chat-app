## What Changed

### 1. The high-risk maintainability boundaries were substantially thinned

Recent work has already moved the repo much closer to stable boundaries:

- `sendToGptFlow` now routes request, finalize, response, and protocol-block shaping through dedicated builders
- `app/page.tsx` now composes grouped page domains instead of owning most wiring inline
- task runtime / draft sync / Kin transfer handoff now uses shared builders
- ingest authority is much closer to one shared canonical-document model
- server routes now follow a much more consistent thin-entrypoint pattern
- responsive single-panel heuristics now flow through shared viewport/device helpers
- fallback debug payload is no longer persisted into memory state, and the obsolete `gptMemoryStateSummaryMerge.ts` branch has been removed so memory compaction now follows one authoritative path
- `lib/memory.ts` now declares task-scoped memory keys directly, and `gptMemoryStorage` now clears task-scoped state through that shared lifecycle helper instead of a private local cleanup rule
- `useMemoryInterpreterSettings.ts` now persists through `lib/app/memoryRuleStore.ts`, so the memory rule system no longer spreads raw localStorage parsing/writing across the hook itself
- `useGptMemory.ts` now delegates runtime load/update/reapply orchestration through `lib/app/gptMemoryRuntime.ts`, leaving the hook closer to a lifecycle facade and moving the runtime handoff behind a tested boundary
- `docs/memory-lifecycle.md` now fixes the intended split between stable memory, task-scoped memory, and displayed-context memory, and `lib/memory.ts` exposes matching lifecycle key lists
- token accounting now restores one authoritative total-token aggregate, includes memory compaction usage inside conversation recent/cumulative totals, relabels the old summary line as compaction, and counts ingest-time summary generation usage under ingest instead of conversation compaction

This means future work should be much more conservative than the earlier structural passes.

### 2. The main remaining risks are regrowth and half-removed compatibility paths

Instead, the highest-signal risks are:

## Operational Rules For The Next Session

### 1. Do not trust old handoffs over green verification

If a dated handoff disagrees with:

- `README.md`
- `docs/refactor-roadmap.md`
- `docs/next-session.md`
- fresh verification results

then update the handoff before using it to drive priorities.

### 2. Treat builder files as dangerous if they start making decisions

Builder modules are allowed to own:

- grouping
- reshaping
- field mapping

They should not quietly own:

- policy
- behavioral branching
- fallback business rules

### 3. Prefer "one structural boundary per sprint"

The repo is now healthy enough that broad rescue refactors are no longer the default.

Prefer:

1. one boundary
2. one test set
3. one doc sync pass

instead of reopening many unrelated areas together.

### 4. Use the maintenance checklist before claiming completion

Do not say maintenance is complete from intuition alone.

Use:

- `docs/maintenance-checklist.md`

and update the shared status docs together when a checklist item changes.

### 5. LLM regressions must be debugged end-to-end

Recent regressions showed that it is not enough to inspect only the helper most
recently edited.

Before saying that an old overwrite/rewrite path is gone, inspect:

1. prompt sent to the model
2. raw model reply
3. parsed reply
4. every local post-parse transform
5. the final adopted value in UI/state

The preferred fix style is subtractive:

- remove the interfering rewrite at its root
- do not add prompt inflation, new guards, or compensating rules unless the
  root cause is already proven and still needed

### 6. Be explicit about uncertainty

Recent failures were worsened by treating partial inspection as confirmation.

Operational rule:

- separate observed runtime facts from working hypotheses
- if only nearby code was inspected, say so
- do not say a legacy path is gone unless the full runtime path proved it
