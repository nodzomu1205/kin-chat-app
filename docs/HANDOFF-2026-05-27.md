# Handoff 2026-05-27

## Scope

This closeout covers the Kin panel UI cleanup and GPT library settings UI
cleanup completed after the multi-recipient Kin send work.

## Kin Panel UI

- `Kin間チャット` now appears as a hanging tab below the Kin header, matching the
  GPT settings tab style more closely than the old header pill.
- Kin drawer panels are mutually exclusive: opening Kin list, connection, or
  Kin-to-Kin chat closes the other open Kin drawer.
- Kin-to-Kin chat controls now fit within the mobile viewport instead of
  relying on horizontal scrolling.
- The Kin-to-Kin status pill sits next to the `Kin-to-Kin chat` title.
- Kin list cards are more compact:
  - Kin ID is shown on the `送信対象` row.
  - the display-name edit field sits on the Kin name/action row.
  - the edit field is intentionally short on desktop and full-width only when
    mobile wrapping requires it.

## GPT Library Settings UI

- Library settings now display in this order:
  1. direct library reference plus library capacity stats
  2. image library reference
  3. DB reference
- The old separate library-capacity panel was merged into direct reference.
- Panel-level helper descriptions and redundant panel titles were removed to
  reduce vertical space.
- User-facing `RAG参照` labels in the settings UI were renamed to `DB参照`.

## DB Candidate Chunk Limit

- `DB参照 検索候補チャンク上限` now accepts `0`.
- The field uses committed editing so users can delete the current number and
  re-enter another value.
- `0` persists and reloads as a valid setting.
- When the candidate chunk limit is `0`, DB candidate search is skipped with
  `candidate_count_zero` rather than being silently rounded back to `1`.
- The UI prop builder, browser search client, API validation, and server search
  boundary all accept the same `0` behavior.

## Kin Outgoing Message Limit

- Kin-facing protocol prompts now tell Kin to keep each outgoing message at or
  under 500 characters.
- If Kin output would exceed 500 characters, prompts now instruct Kin to split
  it into 400-500 character parts, labeled as `PART n/total`, with the final
  part clearly marked.
- The same limit is shared by the default protocol prompt, protocol rulebook,
  compiled `SYS_TASK` delivery limits, and automatic resend guidance after an
  unusable Kin response.
- Existing saved 700-character protocol prompts are migrated to the 500 /
  400-500 rule through `migrateLegacyProtocolLimits(...)`.
- The active limit values live in `lib/shared/kinMessageLimits.ts`; keep future
  Kin outgoing message limit text routed through that shared source.

## GPT Chat Toolbar Cleanup

- The GPT chat tab toolbar now labels `translate_explain` as `解説`.
- The old `翻訳` label is no longer used in the GPT chat toolbar.
- The GPT chat tab toolbar now labels `reply_only` as `返信案`.
- The old `返信のみ` label is no longer used in the GPT chat toolbar.
- The old `整える` toolbar button and its `polish` instruction mode were removed
  from the UI type, toolbar rendering, server request normalization, and prompt
  builder path.

## Explanation Reply-Draft Flow

- `解説` output now uses this fixed structure:

```text
[原文]
[日本語訳]
[解説]

返信案を作りますか？
```

- `[解説]` is intentionally brief. Do not expand it back into detailed linguistic
  analysis unless the user explicitly asks for that behavior.
- When the user answers the reply-draft offer with a normal affirmative reply
  such as `はい。` or `はい。短く回答して下さい。`, the server rewrites that final
  request into a reply-draft request using the immediately preceding explained
  source message as context.
- Reply drafts from this flow must use the same language as the original source
  message in `[原文]`.
- The affirmative follow-up detector lives in
  `lib/shared/replyDraftFollowup.ts` so client-side reference selection and
  server-side prompt shaping use the same boundary.
- Reply-draft follow-ups intentionally skip DB/RAG reference lookup and direct
  library reference injection. The flow only needs the immediate chat history;
  running the DB/library reference path made short `はい。` follow-ups
  impractically slow.
