## Completed In This Session

### Facilitator-Led Group Chat

The Kin-to-Kin drawer now allows selecting one or more participant Kin profiles
besides the starter Kin.

Behavior:

- One selected participant keeps the existing two-Kin alternating relay path.
- Two or more selected participants use group mode.
- The initial message is sent only to the starter Kin.
- The starter Kin is the facilitator and chooses the next participant by
  replying with the existing `<<SYS_[Facilitator]_TO_[Participant]_CHAT>>`
  format.
- Participants reply only to the facilitator with
  `<<SYS_[Participant]_TO_[Facilitator]_CHAT>>`.
- Incoming messages are relayed as plain message bodies with short English
  system notices, not re-wrapped in the incoming `SYS_A_TO_B_CHAT` block.
- The app validates each route before relaying.
- Invalid routing or malformed protocol replies trigger a retry notice to the
  same Kin. The session errors after repeated retry failure instead of looping
  forever.
- Limit notices are sent to every selected member when the max count is reached.
- The facilitator can also end group chat early by including
  `**END THE CHAT**` in the message body. The app records that facilitator
  message, then sends the same completion notice to every selected member.
- Participant messages that contain `**END THE CHAT**` do not end the session;
  early termination is facilitator-only.

### Drawer UI Cleanup

The drawer participant selector was tightened after live review:

- the old single partner select was replaced with compact ON/OFF participant
  chips
- one selected participant keeps the original two-Kin mode
- two or more selected participants switch to group mode
- selected participants are summarized in one line
- the participant grid scrolls within a compact height so larger Kin lists do
  not expand the drawer
- PC layout uses left-aligned fixed-width controls instead of stretching every
  field across the available width

### Prompt / Notice Shape

No new turn protocol block was introduced. Group mode still uses the existing
`SYS_[AAA]_TO_[BBB]_CHAT` envelope as the only machine-readable chat message
format.

The new English notices are built in:

- `buildKinGroupChatStartBlock`
- `buildKinGroupChatRelayBlock`
- `buildKinGroupChatRetryBlock`

Routing validation is built around:

- `parseKinToKinProtocolBlock`
- `validateKinGroupChatRoute`
- `containsKinGroupChatEndToken`
