## Completed In This Slice

### Library Operations

- Library operations UI now has compact grouped tiles for:
  - device import
  - Google Drive browse/import
  - ordinary library bulk display / Kin send
  - image-library bulk display / Kin send
- Ordinary library bulk operations keep `Index / Summary / Summary + Detail`.
- Image-library bulk display now includes visible images.
- Image-library bulk Kin send intentionally remains text/metadata only, without
  embedded image binaries.

### Task / Kin Protocol

- Registered task constraints now support ordinary library reference,
  image-library reference, search, and PPT-work counts as separate concepts.
- Task-time library reference settings and task-time image-library reference
  settings are distinct from Kin data-request protocols.
- Kin can request image-library metadata through
  `SYS_LIBRARY_IMAGE_DATA_REQUEST / RESPONSE`.
- Kin can request PPT design work through
  `SYS_PPT_DESIGN_REQUEST / RESPONSE`.
- PPT design work counts separately in task progress as `PPT作業`.
- PPT design responses can later be saved through the existing
  `SYS_FILE_SAVING_REQUEST` path; do not add a second file-save route.

### PPT Direct Edit

- `/ppt` direct edit now uses a dedicated LLM interpretation format
  (`presentation_direct_edit`) instead of regenerating the whole PPT design and
  diffing it.
- The direct-edit interpretation identifies:
  - slide number
  - block number
  - block id
  - actual block type: text or visual
  - user instruction for that block
  - proposed replacement text, or visual replacement/generation fields
- The approval UI lives at the top of Library settings and exposes editable
  fields for target, instruction, and proposed content.
- Approval is required before applying a new direct edit.
- Approved strong matches bypass the LLM next time and apply immediately.
- Approved direct edits update both:
  - the rendered PPTX
  - the saved PPT design document in the library

## Watch Items

### Direct Edit Layout Capacity

Text overflow after a valid replacement is still a real design issue. Do not
solve it with a renderer-wide font-shrink change by default.

Next design should consider:

- how LLM proposed text receives layout capacity hints
- whether direct-edit approval should show a density/fit warning
- whether block-level `renderStyle.textStyle` should be attached only to the
  touched block
- whether a larger requested edit should be redirected to PPT design revision
  instead of direct PPTX edit

### Direct Edit Retry

The approval UI does not yet have the suggested "retry LLM with edited target
fields" flow. The user can edit fields manually and approve, but a re-run button
is still pending.

### Visual Direct Edit

The model supports visual edits structurally:

- library image replacement by Image ID
- semantic image-library replacement
- image generation prompt
- visual brief/prompt revision

Manual testing should cover at least one visual replacement and one generated
visual before calling this path settled.

### Auto Reference

Current intended policy:

- normal chat: ordinary library auto-reference ON/OFF applies
- normal task button flow: ordinary library auto-reference ON/OFF applies
- PPT task button flow: ordinary library and image-library auto-reference
  ON/OFF apply
- Kin PPT design request: task-time ordinary/image-library reference settings
  apply
- explicit data import / response import / search import remain separate
  request paths

Do not merge ordinary library reference and image-library reference into one
concept. They are related projections over stored material, but have different
runtime purposes.
