## Completed In This Session

### DB Reference UX And Candidate Controls

The DB tab is now a real management surface for RAG documents:

- registered DB documents are listed with chunk counts
- DB candidate ordering controls determine which documents are eligible for RAG
  search
- the RAG search RPC receives candidate document ids and searches only inside
  that candidate set
- mobile overflow in the DB tab was corrected
- DB reference logs show actual runtime matches, similarity, and context size

The current RAG reference settings are:

- candidate chunk upper bound
- context chunk upper bound
- similarity threshold

The candidate boundary is ordered by DB document cards. A document can push the
actual included chunk count slightly above the configured upper bound because
the unit of manual ordering is currently a DB document, not an individual
chunk.

### DB Delete

DB document deletion is implemented:

- UI button: `DB delete`
- API route: `DELETE /api/library-rag/documents`
- server helper: `deleteSupabaseRagLibraryDocument`
- Supabase `rag_document_chunks` are deleted by the existing cascade relation

Deletion is explicit and confirmed by the browser prompt.

### Duplicate And Similarity Candidates

The DB tab now surfaces duplicate and merge candidates:

- exact duplicate documents by `content_hash`
- exact duplicate chunks by normalized text
- semantic similar chunk pairs through Supabase `pgvector`

The semantic RPC added to `supabase/rag_library.sql` is:

```sql
public.find_similar_rag_library_chunks(
  min_similarity double precision default 0.68,
  max_pairs integer default 30
)
```

It compares stored chunk embeddings against other chunk embeddings and returns
cross-document chunk pairs. The app treats a missing RPC as a safe empty result,
so old Supabase schema state does not break the DB tab.

Important live observation:

- exact duplicate detection works
- DB deletion works
- semantic similar detection is implemented, but with the current stored data it
  still did not produce useful candidates in the user's live test

This means the next design should not rely only on automatic pairwise similarity
surfacing.

### Non-Destructive Compaction

The DB tab can create a compacted DB document from a candidate group:

- UI action: create integrated DB document
- API route: `POST /api/library-rag/documents`
- server flow: `compactSupabaseRagLibraryDocuments`
- source documents are kept
- selected source chunks are deduplicated locally and re-indexed as a new DB
  document with fresh embeddings

This is intentionally non-destructive. The user can inspect the integrated
document first, then delete old documents manually if desired.

### SerpAPI Search Skip Rule

Pure inline SerpAPI search inputs no longer trigger DB search.

Examples that now skip DB reference:

```text
Search: Tokyo weather
検索: AI これから
検索：AI　これから
```

Inputs with both a search query and additional user instruction can still use DB
reference:

```text
検索: 帰国子女 高校
DB情報も踏まえて比較して
```

Protocol search blocks such as `SYS_SEARCH_REQUEST` remain DB-skip paths.

## Current Limitations

The automatic near-duplicate route is not yet enough for real DB organization.
The user observed no useful semantic candidate output even after switching from
text n-gram similarity to stored embedding similarity.

The likely better next workflow is user-guided organization:

1. extract categories, themes, entities, and document types from current DB
   documents/chunks
2. sort and group DB documents by those extracted labels
3. let the user select multiple documents/chunks manually
4. create an integrated DB document from that selected set
5. keep originals until the user explicitly deletes or supersedes them

This should work better than relying on hidden automatic pair detection because
the user can judge whether two documents are close enough to merge.

## Stop Rules

- Do not auto-delete or auto-merge DB knowledge without user approval.
- Do not treat Library cards as live-synced with DB documents.
- Do not rely solely on hidden pairwise similarity for DB organization.
- Keep DB compaction non-destructive unless the user explicitly chooses a
  destructive delete/supersede action.
- Keep SerpAPI-only search free from DB reference unless the user adds a
  separate instruction that clearly benefits from DB context.
